链表基础

根据*代码随想录*中对链表基础的描述，将其总结如下：

算法题

1、移除链表元素(简单)

题目表述

给你一个链表的头节点 head 和一个整数 val ，请你删除链表中所有满足 Node.val == val 的节点，并返回 新的头节点 。移除链表元素

思考

2023/11/7第二次思考：遍历链表使用while结构，如果只在一个原链表上进行删除操作的话，那么每遍历到需要删除的节点，就要把这节点之后的节点连接到上一节点上。可是这是单向链表，无法从该节点向上溯源。那么，就应该初始化一个空链表，遍历原始链表，将原始链表上的节点添加到空链表上，碰见需要删除的节点就跳过。

题解(自己写的—迭代)

时间复杂度：O(n)

空间复杂度：O(1)

官方题解(递归)

时间复杂度：O(n)

空间复杂度：O(n) 主要取决于递归调用栈，最多不会超过N层。

自己的思考：链表这种线性结构也可以使用递归方法去遍历。那么递归的三要素：确定递归的参数和返回值、终止条件和单层递归的逻辑。递归需要从后往前看，最重要的是确定单层递归的逻辑，如果当前节点是需要删除的，那么就跳过这个节点，返回下一节点。

2、设计链表(中等)

题目表述

你可以选择使用单链表或者双链表，设计并实现自己的链表。

单链表中的节点应该具备两个属性：val 和 next 。val 是当前节点的值，next 是指向下一个节点的指针/引用。

如果是双向链表，则还需要属性 prev 以指示链表中的上一个节点。假设链表中的所有节点下标从 0 开始。

实现 MyLinkedList 类：

  • MyLinkedList() 初始化 MyLinkedList 对象。
  • int get(int index) 获取链表中下标为 index 的节点的值。如果下标无效，则返回 -1 。
  • void addAtHead(int val) 将一个值为 val 的节点插入到链表中第一个元素之前。在插入完成后，新节点会成为链表的第一个节点。
  • void addAtTail(int val) 将一个值为 val 的节点追加到链表中作为链表的最后一个元素。
  • void addAtIndex(int index, int val) 将一个值为 val 的节点插入到链表中下标为 index 的节点之前。如果 index 等于链表的长度，那么该节点会被追加到链表的末尾。如果 index 比长度更大，该节点将 不会插入 到链表中。
  • void deleteAtIndex(int index) 如果下标有效，则删除链表中下标为 index 的节点。

思考

2023/11/8第一次思考：当看到实现的方法中有在尾部和在头部添加节点，那么是否可以维护两个节点这样的节点——一个始终指向头节点，一个始终指向尾节点，这样就能快速实现在头尾部分别插入节点的功能。同样方法实现内容中有get方法用来获取索引下的节点值，那么我们就可以维护一个单向链表长度的属性，用来实现get方法。

具体而言，addAtHead和addAtTail方法都比较容易实现，问题在于如何高效的实现在某一索引的位置上添加或删除节点。

对于deleteAtIndex方法,如果要删除的节点索引≥链表的长度，实际上待删除的节点是不存在的。如果待删除的节点就是首节点，那么直接让head = head->next并且length--直接返回。如果待删除节点在中间范围，那么就要遍历整个链表，由于维护的是链表的首、尾节点，遍历从头遍历，那么就要把首节点存一下，拿首节点进行遍历最终再恢复首节点。另一方面，在删除节点时，会变动尾节点，那么也照样在遍历时把尾节点更新一下。

对于addAtIndex方法，如果插入的节点索引大于链表的长度，实际上也待插入节点的索引逻辑是错误的，也就插不了。如果待插入节点正好是在尾部，那么直接可以调用addAtTail方法实现。如果待插入节点在头部，与deleteAtIndex方法类似。如果待插入节点的索引在中间范围，那么就需要遍历整个链表。

在遍历时保存首节点，在对整个链表进行遍历，head=head->next，在修改时即对前方已经遍历过的节点不造成影响。

题解(自己写的)

官方题解(双向链表)

时间复杂度：初始化消耗*O(1)*，get 消耗 *O(index)*，addAtHead和addAtTail消耗 *O(1)*，addAtIndex消耗 *O(index)*。

空间复杂度：所有函数单词调用的空间复杂度均为O(1)，总体空间复杂度为O(n)，n为addAtHead、addAtTail和addAtIndex调用次数之和。

3、反转链表(简单)

题目表述

给你单链表的头节点 head ，请你反转链表，并返回反转后的链表。

思考

2023/11/8第二次思考，修改链表第一眼想到的就是使用迭代方法遍历链表，但是在遍历的过程中是将链表的每个节点指向修改，那么不免就会打破链表的链子，链子一旦断掉那么无法继续遍历。于是，需要在修改当前节点的指向时，先把当前节点指向的下一节点先存起来，完事之后再修改当前节点的指向，其指向上一节点，那么上一节点pre也需要在前一代修改时保存。修改完成之后，需要把遍历的链表再接上继续遍历，如此完成整个实现过程。

题解(自己写的——迭代法)

时间复杂度：O(n)，需要遍历链表一次。

空间复杂度：O(1)

官方题解（递归）

时间复杂度：O(n)，其中 n是链表的长度。需要对链表的每个节点进行反转操作。

空间复杂度：O(n)，其中 n 是链表的长度。空间复杂度主要取决于递归调用的栈空间，最多为 n 层。

自己的思考：该方法的递归版本难想一些，因为需要反转链表，假设链表的其余部分已经被反转，那么应该如何反转该节点之前的部分？
